## 2. 系统架构设计

### 2.1 架构概述

```
┌──────────────────────────────────────────────────────────┐
│                      浏览器客户端                         │
│              (HTML + CSS + JavaScript)                   │
│              public/*.html, public/js/api.js             │
└──────────────────────┬───────────────────────────────────┘
                       │ HTTP (JSON / CSV)
                       ▼
┌──────────────────────────────────────────────────────────┐
│                  Express 应用服务器                        │
│                   server/index.js                        │
│                  监听端口: 3000                           │
│                                                          │
│  ┌──────────────┐  ┌──────────────────────────────────┐ │
│  │  中间件层     │  │  路由层 (7 个模块)               │ │
│  │              │  │                                  │ │
│  │  cors        │  │  /api/auth      → auth.js       │ │
│  │  express.json│  │  /api/families   → families.js   │ │
│  │  authMiddleware│  │  /api/members    → members.js    │ │
│  │  requireRole │  │  /api/records    → records.js    │ │
│  │              │  │  /api/asset-types→ asset-types.js│ │
│  │              │  │  /api/stats      → stats.js      │ │
│  │              │  │  /api/export     → export.js     │ │
│  │              │  │                                  │ │
│  │              │  │  express.static  → public/       │ │
│  └──────┬───────┘  └──────────────┬───────────────────┘ │
│         │                         │                      │
└─────────┼─────────────────────────┼──────────────────────┘
          │                         │
          ▼                         ▼
┌──────────────────────────────────────────────────────────┐
│              better-sqlite3 (同步 SQLite 驱动)            │
│                   server/db.js                           │
│              WAL 模式 | 参数化查询                        │
└──────────────────────┬───────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────┐
│              SQLite 数据库文件                             │
│              server/data/users.db                        │
│              5 张数据表                                   │
└──────────────────────────────────────────────────────────┘
```

**架构特点**：
- **单体架构**：前后端一体部署，Express 同时提供 API 和静态文件服务
- **同步数据库驱动**：使用 better-sqlite3 的同步 API，无需 async/await，代码简洁
- **WAL 模式**：SQLite 启用 WAL（Write-Ahead Logging），提升并发读写性能

### 2.2 技术选型

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|------|----------|
| 运行时 | Node.js | - | JavaScript 全栈，前后端语言统一 |
| Web 框架 | Express | ^5.2.1 | 轻量、成熟、生态丰富 |
| 数据库 | SQLite (better-sqlite3) | ^12.10.0 | 零配置、嵌入式、适合单机部署 |
| 密码加密 | bcryptjs | ^3.0.3 | 纯 JS 实现，无需编译原生模块 |
| 认证令牌 | jsonwebtoken | ^9.0.3 | JWT 标准，无状态认证 |
| 跨域 | cors | ^2.8.6 | Express 标准跨域中间件 |
| Excel 导出 | exceljs | ^4.4.0 | 支持 .xlsx 格式读写 |
| 文件上传 | multer | ^1.4.5-lts.1 | Express 标准文件上传中间件 |
| 单元测试 | Jest | ^30.4.2 | 零配置，快照测试 |
| API 测试 | Supertest | ^7.2.2 | HTTP 断言，不启动真实端口 |
| E2E 测试 | Playwright | ^1.60.0 | 多浏览器支持，自动等待 |

## 4. 数据库设计

### 4.1 数据库概述

- **数据库类型**: SQLite 3
- **数据库文件**: `server/data/users.db`
- **写入模式**: WAL (Write-Ahead Logging)
- **字符编码**: UTF-8
- **时间格式**: ISO 8601 字符串（`datetime('now', 'localtime')`）
- **主键策略**: INTEGER PRIMARY KEY AUTOINCREMENT（自增整数）

### 4.2 实体关系图

```
┌──────────────┐       ┌──────────────┐       ┌──────────────┐
│   families   │       │   members    │       │    users     │
├──────────────┤       ├──────────────┤       ├──────────────┤
│ id (PK)      │──┐    │ id (PK)      │       │ id (PK)      │
│ family_name  │  │    │ family_id(FK)│◄──────│ username     │
│ head_id (FK) │──┼───►│ name         │       │ password     │
│ created_at   │  │    │ short_name   │       │ role         │
└──────────────┘  │    │ role         │       │ member_id(FK)│──┐
                  │    │ created_at   │       │ created_at   │  │
                  │    └──────────────┘       └──────────────┘  │
                  │           │                                 │
                  │           │ 1:N                             │
                  │           ▼                                 │
                  │    ┌──────────────┐                         │
                  │    │   records    │                         │
                  │    ├──────────────┤                         │
                  └───►│ id (PK)      │                         │
                       │ member_id(FK)│◄────────────────────────┘
                       │ family_id(FK)│
                       │ type         │──────┐
                       │ name         │      │
                       │ value        │      │
                       │ previous_val │      │
                       │ date         │      │
                       │ status       │      │
                       │ note         │      │
                       │ created_at   │      │
                       └──────────────┘      │
                                             │
                       ┌──────────────┐      │
                       │ asset_types  │      │
                       ├──────────────┤      │
                       │ id (PK)      │      │
                       │ type_value   │◄─────┘
                       │ display_name │
                       │ color        │
                       │ is_custom    │
                       │ family_id(FK)│
                       │ created_at   │
                       └──────────────┘
```

**关系说明**:
- `families` 1 : N `members`（一个家庭包含多个成员）
- `families.head_id` → `members.id`（户主指向成员）
- `members` 1 : N `records`（一个成员拥有多条资产记录）
- `records.type` → `asset_types.type_value`（资产记录引用资产类型）
- `users.member_id` → `members.id`（用户账户关联成员，admin 为 NULL）

### 4.3 数据表详细设计

#### 4.3.1 users（用户账户表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | 用户唯一标识 |
| `username` | TEXT | UNIQUE NOT NULL | 登录用户名 |
| `password` | TEXT | NOT NULL | bcrypt 加密后的密码哈希（saltRounds=10） |
| `role` | TEXT | NOT NULL, DEFAULT 'member' | 角色：`admin` / `head` / `member` |
| `member_id` | INTEGER | FOREIGN KEY → members(id) | 关联的家庭成员，admin 为 NULL |
| `created_at` | TEXT | NOT NULL, DEFAULT datetime('now','localtime') | 创建时间 |

```sql
CREATE TABLE IF NOT EXISTS users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  username TEXT UNIQUE NOT NULL,
  password TEXT NOT NULL,
  role TEXT NOT NULL DEFAULT 'member',
  member_id INTEGER,
  created_at TEXT NOT NULL DEFAULT (datetime('now', 'localtime'))
);
```

#### 4.3.2 families（家庭信息表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | 家庭唯一标识 |
| `family_name` | TEXT | NOT NULL | 家庭名称 |
| `head_id` | INTEGER | FOREIGN KEY → members(id) | 户主成员 ID |
| `created_at` | TEXT | NOT NULL, DEFAULT datetime('now','localtime') | 创建时间 |

```sql
CREATE TABLE IF NOT EXISTS families (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  family_name TEXT NOT NULL,
  head_id INTEGER,
  created_at TEXT NOT NULL DEFAULT (datetime('now', 'localtime')),
  FOREIGN KEY (head_id) REFERENCES members(id)
);
```

#### 4.3.3 members（家庭成员表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | 成员唯一标识 |
| `family_id` | INTEGER | NOT NULL, FOREIGN KEY → families(id) | 所属家庭 ID |
| `name` | TEXT | NOT NULL | 成员姓名 |
| `short_name` | TEXT | - | 简称，用于头像显示（默认取 name 首字符） |
| `role` | TEXT | NOT NULL, DEFAULT 'member' | 成员角色：`head`（户主）/ `member`（普通成员） |
| `created_at` | TEXT | NOT NULL, DEFAULT datetime('now','localtime') | 创建时间 |

```sql
CREATE TABLE IF NOT EXISTS members (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  family_id INTEGER NOT NULL,
  name TEXT NOT NULL,
  short_name TEXT,
  role TEXT NOT NULL DEFAULT 'member',
  created_at TEXT NOT NULL DEFAULT (datetime('now', 'localtime')),
  FOREIGN KEY (family_id) REFERENCES families(id)
);
```

#### 4.3.4 asset_types（资产类型表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | 类型唯一标识 |
| `type_value` | TEXT | UNIQUE NOT NULL | 类型标识（英文 key，如 `stock`、`fund`） |
| `display_name` | TEXT | NOT NULL | 显示名称（如"股票"、"基金"） |
| `color` | TEXT | NOT NULL | 颜色值（十六进制，如 `#f59e0b`） |
| `is_custom` | INTEGER | DEFAULT 0 | 0 = 系统预设类型，1 = 家庭自定义类型 |
| `family_id` | INTEGER | FOREIGN KEY → families(id) | 自定义类型所属家庭（系统预设为 NULL） |
| `created_at` | TEXT | NOT NULL, DEFAULT datetime('now','localtime') | 创建时间 |

```sql
CREATE TABLE IF NOT EXISTS asset_types (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  type_value TEXT UNIQUE NOT NULL,
  display_name TEXT NOT NULL,
  color TEXT NOT NULL,
  is_custom INTEGER DEFAULT 0,
  family_id INTEGER,
  created_at TEXT NOT NULL DEFAULT (datetime('now', 'localtime')),
  FOREIGN KEY (family_id) REFERENCES families(id)
);
```

#### 4.3.5 records（资产记录表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | 记录唯一标识 |
| `member_id` | INTEGER | NOT NULL, FOREIGN KEY → members(id) | 关联家庭成员 ID |
| `family_id` | INTEGER | NOT NULL, FOREIGN KEY → families(id) | 所属家庭 ID（冗余字段，加速查询） |
| `type` | TEXT | NOT NULL | 资产类型（对应 asset_types.type_value） |
| `name` | TEXT | NOT NULL | 资产名称（如"招商银行股票"） |
| `value` | REAL | NOT NULL | 当前价值（单位：元） |
| `previous_value` | REAL | - | 上期价值，用于计算涨跌（NULL 表示无上期数据） |
| `date` | TEXT | NOT NULL | 记录日期（格式：YYYY-MM-DD） |
| `status` | TEXT | NOT NULL, DEFAULT 'valid' | 状态：`valid`（有效）/ `pending`（待审核）/ `invalid`（无效） |
| `note` | TEXT | - | 备注说明 |
| `created_at` | TEXT | NOT NULL, DEFAULT datetime('now','localtime') | 创建时间 |

```sql
CREATE TABLE IF NOT EXISTS records (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  member_id INTEGER NOT NULL,
  family_id INTEGER NOT NULL,
  type TEXT NOT NULL,
  name TEXT NOT NULL,
  value REAL NOT NULL,
  previous_value REAL,
  date TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'valid',
  note TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now', 'localtime')),
  FOREIGN KEY (member_id) REFERENCES members(id),
  FOREIGN KEY (family_id) REFERENCES families(id)
);
```

## 5. API 接口规范

### 5.1 通用约定

#### 5.1.1 基础信息

| 项目 | 说明 |
|------|------|
| 协议 | HTTP/1.1 |
| 数据格式 | JSON（导出接口为 CSV） |
| 字符编码 | UTF-8 |
| 端口 | 3000 |
| 基础路径 | `/api` |

#### 5.1.2 认证方式

除 `/api/auth/login` 和 `/api/auth/logout` 外，所有业务接口均需 JWT 认证。

请求头：
```
Authorization: Bearer <token>
```

Token 获取方式：`POST /api/auth/login` 成功后返回。

#### 5.1.3 统一响应格式

**成功响应**：
```json
{
  "success": true,
  "data": { ... },
  "message": "操作成功"
}
```

**错误响应**：
```json
{
  "success": false,
  "message": "错误描述"
}
```

#### 5.1.4 HTTP 状态码

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|

### 5.1.6 数据类型处理规范

**问题诊断记录**：
- 问题现象：成员账户添加资产记录时，提示"成员不存在"
- 根本原因：HTML select.value返回字符串类型，SQLite查询时类型不匹配
- 解决方案：前后端都显式进行类型转换

**前端数据发送规范**：

1. **memberId**：必须发送为数字类型（Number）
   ```javascript
   // ✅ 正确
   memberId: 2
   
   // ❌ 错误（会导致后端查询失败）
   memberId: "2"
   ```

2. **value/previousValue**：必须发送为数字类型
3. **date**：必须发送为字符串格式 "YYYY-MM-DD"

**前端转换方法**：
```javascript
// 从 select 元素获取值时转换为数字
const memberId = parseInt(selectElement.value, 10);

// 确保值是数字类型
if (typeof memberId === 'string') {
    memberId = parseInt(memberId, 10);
}
```

**后端数据接收规范**：

1. **后端兼容性处理**：
   ```javascript
   // 确保 targetMemberId 是数字类型
   const numericMemberId = typeof targetMemberId === 'string' 
       ? parseInt(targetMemberId, 10) 
       : targetMemberId;
   ```

2. **所有ID类字段**：统一转换为数字类型后再进行数据库查询

**重要提示**：
- HTML select 元素的 value 属性始终是字符串类型
- 必须通过 `parseInt()` 或 `Number()` 显式转换
- 前后端都应该做类型检查，确保数据一致性

### 5.6.1 数据库事务最佳实践

**何时使用事务**:
1. 涉及多个写操作（INSERT/UPDATE/DELETE）的业务流程
2. 需要保证操作原子性，避免数据不一致的场景
3. 恢复默认类型等批量操作

**事务模式**:
```javascript
db.prepare('BEGIN TRANSACTION').run();
try {
  // 业务逻辑...
  db.prepare('COMMIT').run();
  res.json({ success: true });
} catch (error) {
  db.prepare('ROLLBACK').run();
  res.status(500).json({ success: false });
}
```

**已实现事务保护的接口**:
1. DELETE `/api/asset-types/:id` - 删除类型时的迁移操作
2. POST `/api/asset-types/restore-default` - 批量恢复默认类型

#### GET `/api/stats` — 获取仪表盘统计数据

**权限控制**:
- admin：全局统计
- head：本家庭统计
- member：个人统计

**Response (200)**:
```json
{
  "success": true,
  "data": {
    "totalValue": 6490000,
    "totalChange": 560000,
    "totalRate": 9.4,
    "totalRecords": 5,
    "memberCount": 3,
    "activeMembers": 3,
    "monthlyNew": 5,
    "pendingCount": 0,
    "typeDistribution": [
      {
        "type": "stock",
        "display_name": "股票",
        "color": "#f59e0b",
        "record_count": 2,
        "total_value": 3550000
      },
      {
        "type": "fund",
        "display_name": "基金",
        "color": "#3b82f6",
        "record_count": 2,
        "total_value": 1740000
      }
    ]
  }
}
```

**字段说明**:

| 字段 | 说明 | 计算方式 |
|------|------|----------|
| `totalValue` | **有效资产总价值（按名称去重取最新）** | 先按 `name` 分组取每组 `date` 最大的记录，再对这些最新记录的 `value` 求和。逻辑：`SUM(latest.value)` FROM (SELECT value FROM records WHERE status='valid' GROUP BY name ORDER BY date DESC LIMIT 1) |
| `totalRecords` | 总记录数 | COUNT(*) |
| `memberCount` | 成员总数 | COUNT(*) FROM members |
| `activeMembers` | 活跃成员数 | 有 status='valid' 记录的成员数 |
| `monthlyNew` | 本月新增记录 | COUNT(*) WHERE date >= 当月1日 |
| `pendingCount` | 待处理记录 | COUNT(*) WHERE status='pending' |
| `typeDistribution` | **按资产类型分布（按名称去重取最新）** | 先按 `type, name` 分组取每组 `date` 最大的记录，再按 `type` 汇总 `record_count` 和 `total_value` |

> ⚠️ **v2.4 变更**: `totalValue` 和 `typeDistribution` 的计算逻辑已修正。原逻辑为 `SUM(value)` 汇总所有历史记录，现改为按资产名称去重，仅统计每个资产名称的最新一期记录。这是因为资产可能不定期更新（有变动才更新），最新记录代表当前真实价值，历史记录用于追溯分析而非当前资产总额计算。

## 6. 认证与授权设计

### 6.1 JWT 认证方案

#### 6.1.1 Token 结构

**签名算法**: HS256 (HMAC with SHA-256)

**密钥**: `ricky_finance_jwt_secret_2024`

**有效期**: 24 小时

**Payload 结构**:
```json
{
  "id": 2,
  "username": "head",
  "role": "head",
  "memberId": 1,
  "name": "张三",
  "familyId": 1,
  "shortName": "张",
  "memberRole": "head",
  "iat": 1716800000,
  "exp": 1716886400
}
```

- `memberId` / `name` / `familyId` / `shortName` / `memberRole`：仅当用户有关联成员时包含，admin 为 null

#### 6.1.2 Token 传递方式

1. **推荐方式**: HTTP Header — `Authorization: Bearer <token>`
2. **备用方式**: URL Query — `?token=<token>`（仅 CSV 导出等 iframe 场景使用）

#### 6.1.3 Token 生成流程

```
用户登录 → findUserByUsername(username) → verifyPassword(password, hash)
→ findUserWithMember(userId) → 组装 payload → jwt.sign(payload, secret, {expiresIn: '24h'})
→ 返回 token + user 信息
```

#### 6.1.4 Token 验证流程

```
请求到达 → 提取 Bearer Token 或 query.token
→ jwt.verify(token, secret) → 解码 payload
→ findUserWithMember(decoded.id) 确认用户仍存在
→ 挂载到 req.user → 进入路由处理
```

### 6.2 中间件设计

#### 6.2.1 authMiddleware（强制认证）

**文件**: `server/middleware/auth.js`

**行为**:
1. 从 `Authorization: Bearer <token>` 或 `?token=` 提取 Token
2. 无 Token → 401 "未登录或登录已过期"
3. JWT 验证 → 解码 payload
4. 从数据库重新查询用户（`findUserWithMember`）
5. 用户不存在 → 401 "用户不存在"
6. Token 过期或无效 → 401 "登录已过期，请重新登录"
7. 挂载 `req.user` → `next()`

**`req.user` 包含字段**:
```
id, username, role, memberId, name, familyId, shortName, memberRole
```

#### 6.2.2 optionalAuth（可选认证）

**行为**:
1. 有 Token → 尝试验证并挂载 `req.user`
2. 无 Token 或 Token 无效 → 静默跳过，`req.user` 为 `undefined`
3. 总是调用 `next()`

#### 6.2.3 requireRole（角色守卫）

**用法**: `requireRole('admin')` 或 `requireRole('head', 'admin')`

**行为**:
- `req.user.role` 不在允许列表中 → 403 "权限不足"

### 6.3 密码安全策略

| 策略 | 实现 |
|------|------|
| 加密算法 | bcrypt (bcryptjs) |
| Salt Rounds | 10 |
| 存储格式 | bcrypt hash 字符串（$2a$10$...） |
| 验证方式 | `bcrypt.compareSync(明文, hash)` |
| 密码长度限制 | 无（依赖 bcrypt 自动处理） |
| 登录失败处理 | 当前无限制（建议后续添加失败次数限制） |

## 7. 安全设计

### 7.1 SQL 注入防护

**策略**: 全部使用参数化查询（Prepared Statements with `?` 占位符）。

**示例**:
```javascript
// ✅ 安全：参数化查询
db.prepare('SELECT * FROM records WHERE member_id = ? AND type = ?').get(memberId, type);

// ❌ 禁止：字符串拼接
db.prepare(`SELECT * FROM records WHERE member_id = ${memberId}`).get();
```

**动态 WHERE 子句构建**:
- 筛选条件通过数组 `whereClauses` 和 `params` 分别收集
- 最终使用 `db.prepare(sql).all(...params)` 安全执行

### 7.2 CORS 策略

当前配置：`app.use(cors())` — 允许所有来源跨域访问。

> ⚠️ **安全建议**: 生产环境应限制为特定域名，如 `cors({ origin: 'https://your-domain.com' })`。

### 7.3 数据隔离策略

**三层数据隔离**（通过 WHERE 条件实现）：

| 角色 | 数据范围 | 实现方式 |
|------|---------|----------|
| admin | 全局数据 | 不添加额外 WHERE 条件 |
| head | 本家庭数据 | `WHERE family_id = req.user.familyId` |
| member | 个人数据 | `WHERE member_id = req.user.memberId` |

**权限二次校验**（在更新/删除操作中）:
- 查询目标记录 → 校验 `family_id` 或 `member_id` 是否与当前用户匹配
- admin 在 records 的增删改操作中直接返回 403
